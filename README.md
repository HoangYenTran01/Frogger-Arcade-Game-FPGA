## About the Project

This project was developed as the final project for **ENG EC 311 – Introduction to Logic Design** at Boston University (December 2025). The goal was to replicate the core gameplay of the 1981 Frogger arcade game entirely in hardware using Verilog HDL, synthesized and deployed on a physical FPGA.

The game runs at 640×480 resolution over VGA at 60 Hz and features moving obstacles, sprite-based graphics, collision detection, score/timer tracking, and multiple game states (intro screen, active gameplay, game over).

**Hardware Target:** Xilinx Artix-7 `xc7a100tcsg324-1` (Digilent Nexys 4 DDR)  
**Toolchain:** Xilinx Vivado Design Suite  
**Language:** Verilog HDL

## Features

- 🎮 **Faithful Frogger gameplay** — cars, logs, turtles, lily pads, lives, score, and countdown timer
- 📺 **VGA display at 640×480 @ 60 Hz** — smooth, glitch-free rendering
- 🐸 **Sprite-based graphics** — all artwork hand-drawn in MS Paint, converted to `.mem` files via a custom Python script
- 🔄 **Smooth object wrapping** — vehicles and logs wrap seamlessly across screen boundaries using VGA porch regions
- 💥 **Collision detection** — bounding-box logic classifies safe zones (road, water, lily pad) and triggers appropriate frog behavior
- 🏆 **Score & timer** — BCD counters drive per-digit ROM displays; life bar and lily pad status use one-hot encoding
- 🎬 **Full game state machine** — intro, active play, death animation, and game-over overlay states
- 🔇 **Debounced input** — reliable directional control from FPGA push buttons

## Technical Implementation

### VGA & Rendering

The VGA controller generates a 25 MHz pixel clock from the 100 MHz board clock. Sprites are stored as ROM `.mem` files generated by a Python script that converts MS Paint images to pixel data. Black pixels act as a transparency mask, allowing sprites to composite cleanly over the background without requiring alpha blending in hardware.

A key challenge was eliminating VGA jitter caused by a timing mismatch between the object position update clock and the pixel scan clock (~0.7% rate difference, producing ~5.6 pixels of frame drift). This was resolved by adding a fixed black border around moving sprites, converting the glitchy boundary between two moving layers into a stable boundary between a moving layer and a fixed region, eliminating the race condition entirely.

Full-screen text overlays (intro screen, game over) are stored in **BRAM** rather than distributed LUTs, significantly reducing LUT/flip-flop usage. Since BRAM requires clock synchronization while the VGA module is asynchronous, a D flip-flop buffers incoming pixel data at 100 MHz — the single-cycle delay is imperceptible at a 25 MHz display rate.

### Object Movement

Each row of obstacles moves at a different speed. Speeds are encoded as pixel-per-frame offsets applied during the VGA back-porch interval (while the display is not active), ensuring updates never cause mid-frame tearing. Log rows, for example, use offsets of +3, -3, +4, +2, and -3 pixels per frame depending on the lane. When the frog rides a log, its x-coordinate is offset by the log's speed each frame.

Bit shifts are used in place of multipliers wherever possible to meet timing constraints and reduce LUT usage.

### Collision Detection

Collision is computed as axis-aligned bounding box overlap between the frog and every active obstacle. The frog's row determines which zone is evaluated (road, river, or safe zone), and the overlap result drives the game FSM to trigger a death, a ride, or a safe landing.

### Game Logic FSM

The game state machine manages five states: `IDLE` → `PLAYING` → `FROG_DEAD` → `GAME_OVER` → `RESET`. A 1 Hz clock (derived via clock division) drives both the countdown timer and timed text overlays. Score increments on successful lily pad landings; lives decrement on collisions with cars or drowning.

## Skills Demonstrated

- **Verilog HDL** — combinational logic, sequential logic, parameterized modules
- **FPGA development** — synthesis, implementation, and bitstream generation in Xilinx Vivado
- **VGA protocol** — horizontal/vertical sync, front/back porch timing, pixel clock generation
- **Memory design** — BRAM instantiation, ROM `.mem` file generation, masking strategies for memory efficiency
- **Finite State Machines (FSMs)** — multi-state game logic and display control
- **Clock domain management** — clock division, cross-domain synchronization, DFF buffering
- **Hardware debugging** — schematic inspection in Vivado, signal-width verification, simulation-based verification
- **Sprite rendering** — pixel-level hardware compositing with transparency masking
- **Arithmetic optimization** — replacing multipliers with bit shifts to meet timing
- **Collaborative hardware design** — modular design with agreed interfaces, Git version control
